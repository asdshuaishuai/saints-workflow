# 审查与调度设计草案

## 一、简介
本草案用于在 `saints-workflow` 中实现多角度、动态化的审查体系：
- 引入 `太上老君` 的需求原子化能力，将需求拆解为可交付的原子（atom）。
- 在原子化之后按“院子”划分责任域（domain/院子），并组织“三清会审”从需求、设计、实现多维度进行调研与评审。
- 由 `天道` 定期/动态调度 `西方二圣`（双重审查）对任务进行定期样本审查与最终审定。

目标产出：需求原子清单、耦合度矩阵、开发层级分配、落地设计文档与审查记录。

## 二、关键角色与职责
- `太上老君`：负责需求接收与原子化（将需求拆分为最小可交付单元），输出原子清单与优先级、复杂度、预估工时等元数据。
- `三清会审`（三方会审）：由需求方、设计方、实现方代表组成，针对每个需求院子进行深度会审与调研，形成落地设计。
- `西方双审`：准提道人（代码审查）、接引道人（业务审查）——由 `天道` 调度，按策略执行双重审查。
- `天道`：负责全局调度、监控与异常干预，按规则触发审查、催促或降级处理。

## 三、流程总览
1. 需求接收（来自产品/Issue/PR）
2. 太上老君原子化（拆分成 atoms，并标注 `owner`、`院子`、`priority`、`complexity`、`coupling_keys`）
3. 院子划分（按领域/栈/发布节奏分派院子负责人）
4. 三清会审（需求方/设计方/实现方共同评审并出具落地设计）
5. 生成耦合度矩阵与开发层级（MVP / 后续迭代 / 长期改造）
6. 天道调度西方双审（定期或动态触发）
7. 审查结论：通过 / 反馈 / 回炉（返工）

## 四、需求原子化（太上老君）
- 输入：需求规范（spec）、Issue 描述、需求 Owner。
- 输出（每个 atom）字段示例：

  - id
  - title
  - description
  - owner
  - 院子（domain）
  - priority（P0/P1/P2）
  - complexity（S/M/L）
  - estimate（人天）
  - coupling_keys（与哪些 atoms 产生耦合）
  - acceptance_criteria

- 原子化规则（建议）：
  - 每个 atom 应可单独交付且可测试
  - 优先按风险/价值/实现复杂度排序
  - 对复杂需求，先拆出最小可行子集（MVP-atom）

## 五、院子划分（如何分配）
- 划分维度：业务域、技术栈、数据边界、发布频率。
- 每个院子维护：院子负责人、支持联系人、常见依赖列表、可复用组件清单。
- 示例映射格式（Markdown/CSV）：

| 院子 | 负责人 | 技术栈 | 说明 |
|---|---:|---|---|
| 身份认证 | @auth-owner | backend, oauth | 包含登录、鉴权接口 |

## 六、三清会审（深度会审）
- 参与者：需求代表（提出方）、设计代表（架构/接口）、实现代表（开发/测试）。
- 目标：从需求完整性、设计可实现性、边界条件、测试覆盖、风险点、耦合影响等多个维度得出可执行的落地设计。
- 会审产物：会审纪要（决议）、落地设计文档（包含接口契约、数据模型、错误处理、回退策略）、任务拆分与优先级。
- 会审节奏：
  - 对于普通需求：在原子化后 1 次会审
  - 对于跨院子或高风险需求：多轮会审 + 设计评审会（必要时邀请安全/性能专家）

## 七、审查维度（多角度）
- 需求层面：业务目标、验收标准、边界条件、非功能需求（性能/安全/可用性）
- 设计层面：架构可靠性、接口契约、数据一致性、伸缩与回退方案
- 实现层面：代码质量、测试覆盖、异常处理、部署影响
- 交叉维度：耦合度（依赖表）、版本兼容策略、迁移计划

## 八、耦合度与开发层级
- 耦合度矩阵：行/列为 atom id，值为耦合强度（0/1/2/3），并标注耦合类型（读/写/契约/时序）
- 开发层级建议：
  - Level 1（独立可交付，优先完成）
  - Level 2（有轻微依赖，需协调）
  - Level 3（高耦合，需联合发布或先做兼容层）

## 九、天道调度器（调度西方双审）设计
- 调度触发：定时（Cron）、事件驱动（任务完成、PR 合并、风险标记）、手动触发。
- 调度策略示例：
  - 每 N 分钟采样未审/已完成的 tasks，优先抽取 `priority` 高、`risk` 高 的任务进行双审
  - 对跨院子或耦合高的任务，触发即时审查（push-mode）
- 调度负载分配：按准提道人/接引道人在线能力、历史负载与专家标签分配任务
- Job Payload（建议 JSON）：

```
{
  "task_id": "T-1234",
  "atoms": ["A-1","A-2"],
  "reason": "post-implementation sample review",
  "priority": "P1",
  "trigger": "cron|event|manual"
}
```

## 十、Agent / 命令接口草案
- 新增命令：
  - `/tiandao schedule-review --task T-1234 --mode cron`：由天道排期审查
  - `/saints review --task T-1234 --stage triage|design|implementation`：启动对应会审流程
  - `/saints atomize --spec spec.md --owner @someone`：调用太上老君进行原子化
- Webhook / 回调：审查结束生成 `review-result` 事件，更新任务状态与会审纪要链接

## 十一、产出物与存储位置
- `docs/reviews/{task_id}/`：会审纪要、落地设计文档、耦合矩阵、审查结论
- `data/atoms/{task_id}.json`：原子化输出（机器可读）
